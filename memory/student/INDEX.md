# 学生记忆系统

> 用的越久，越懂这个孩子。

## 文件结构

```
memory/
├── student/                   # 真实学生数据（隐私，不入 git，本地保留）
│   ├── profile.md             ← 学生画像（长期档案，持续更新）
│   ├── knowledge_matrix.md    ← 知识掌握追踪矩阵
│   ├── templates.md           ← 算法模板库（熟练度追踪）
│   ├── error_log.md           ← 错题本（错误原因、复习排期）
│   ├── contest_log.md         ← 比赛记录（成绩、upsolve、心态）
│   └── conversation_log.md    ← 对话记录（按需归档）
└── student.example/           # 干净空模板（入 git，供新用户参考/复制）
```

> 首次使用：把 `student.example/` 复制为 `student/`，再开始填写真实数据。

## 更新时序与触发（重要）

记忆更新发生在**回复学生之后**，不是之前——你先得知道这轮聊了什么，才有东西可记。但要在输出回复的同一次会话里完成，不要拖到下一次。

### 分级触发：不是每次都要写

| 学生行为 | 更新文件 |
|---------|---------|
| 问/讨论了知识点或算法 | `knowledge_matrix.md` |
| 做了题或练习 | `knowledge_matrix.md` + `templates.md` |
| 做错/卡住/求助 | `error_log.md`（+ 必要时 `templates.md`） |
| 透露个人信息（年级、目标、风格） | `profile.md` |
| 提到比赛/模拟赛 | `contest_log.md` |
| 有学习价值的会话要点 | `conversation_log.md` |

**寒暄、确认、纯粹的"谢谢/在吗"不触发任何更新。** 别为没有学习价值的内容烧读写。

如果有价值的信号同时命中多类，就更新多个文件——按需，不为流程而流程。

## 检索与归档原则

记忆文件会越积越大，按这两条原则避免 context 爆炸：

1. **按需检索，不全量读。** 回复前只读与当前话题相关的文件，且优先读最近条目 / 相关条目，不要把整个 `conversation_log.md` 全文塞进来。
2. **定期归档。** `conversation_log.md` 每个 N 条（或按月）做一次压缩：把旧条目合并成一段月度摘要归入文件末尾「归档」节，正文只保留近期。`error_log.md` 里已标 🟢 已掌握的条目可择期归档。归档不等同删除，是浓缩。

## 诊断逻辑

根据积累的数据，判断引导方向：

```
同一知识点反复出错 → 这块要重新讲思路，不是多刷题能解决
正确率持续上升 → 在进步，维持节奏，适当上难度
突然大量出错 → 先问状态（休息够吗、压力大吗），再谈技术
长时间没碰某知识点 → 主动安排一次复习
```

## 画像深度

随着对话增多，画像由浅入深：

- **基础层**（前几次）：姓名、年级、目标、当前水平
- **特征层**（5-10 次）：学习风格、性格、优势薄弱分布
- **模式层**（20+ 次）：什么场景易粗心、什么题最能激发兴趣、哪种解释方式最有效
- **预测层**（50+ 次）：预判会卡在哪类题、比赛心态变化曲线
